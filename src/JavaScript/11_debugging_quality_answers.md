# デバッグと品質（解答例）
1. 重大度に応じて`error`> `warn`> `info/log`。ログレベルでフィルタしやすくする。
2. `console.table(users)`で配列を表形式に表示。
3. `console.dir`はオブジェクトのプロパティを展開表示。DOM要素は`log`だとHTML、`dir`だとオブジェクトとして見られる。
4. `console.group('task'); ...; console.groupEnd();`で階層化し読みやすくする。
5. `console.time('t'); doWork(); console.timeEnd('t');`
6. 条件が偽のときだけメッセージを出す。`console.assert(x > 0, 'x must be > 0');`
7. ソースマップは変換前の元コード位置をデバッガで復元し、ステップ実行やブレークを可能にする。
8. `debugger`で実行が停止し、開発者ツールが起動してステップ実行できる。
9. 条件付きBPは特定条件でだけ止まり、ループ内の特定ケース調査に便利。
10. ウォッチ式で特定変数を監視し、スコープペインで現在の環境の値を確認する。
11. ネットワークパネルでURL/メソッド/ステータス、リクエスト/レスポンスヘッダー、ペイロードやタイミングを確認。
12. パフォーマンスパネルでLayout/Recalculate Styleの頻発や長いタスクを確認し、レイアウトスラッシングを特定。
13. メモリパネルでヒープスナップショットを比較し、Detached DOMや増え続けるオブジェクトを確認。
14. strict modeでは暗黙グローバルや重複プロパティ等がエラーになる。
15. 関数内`this`が`undefined`になる（非strictでは`window`）。安全性向上のため。
16. ESLintは静的解析でコード規約・バグを検出し、品質と一貫性を高める。
17. `// eslint-disable-next-line rule`等。乱用すると規約が形骸化するので最小限に。
18. Prettierは整形専用、ESLintは構文/バグ検出。役割を分担する。
19. 自動整形でスタイル議論を排除し、レビューをロジック確認に集中できる。
20. 統一規約は可読性向上とバグ混入リスク低減につながる。
21. 単体テストは振る舞いを固定し、リファクタで破壊しても検知できる。
22. テストは動作バグ検出に強い、静的解析は未定義変数や型不一致など構造的バグに強い。
23. ログ粒度は運用で絞れるよう設計し、過剰ログや不足を避ける。
24. コンソールに秘密情報を出すと漏えいリスク。マスクや除外を徹底する。
25. CIでLint/テストを自動化し、人為ミスを減らし、回帰を早期検知できる。
26. フラグや環境変数で挙動を切り替える際、デッドコード除去や設定ミスに注意。
27. スクリーンショットやブラウザ/OS/URLなどを記録すると再現性とデバッグ効率が上がる。
28. `try/catch`でユーザー通知しつつ`console.error`や監視ツールへ送るのが基本。
29. JSDocやTypeScriptは型チェックで早期に不整合を検出し、補完性も向上する。
30. 例: 「Lint/テストを走らせる」「秘密情報ログ出力禁止」「例外を握り潰さない」。毎コミット前に確認する習慣を付ける。
